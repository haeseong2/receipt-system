✅ 프로젝트 전체 진행 흐름 (최종 정리본)
1️⃣ 전체 폴더 구조 전면 정리
프론트 / 백엔드 / OCR 폴더 확정
pages / components / service / dto / domain 등 뼈대만 생성
코드 구현은 최소화, 구조 선언용

목적:
→ 프로젝트 아키텍처와 책임 경계 고정

2️⃣ OCR 교체 → PaddleOCR 적용
PaddleOCR로 실제 영수증 다수 테스트
텍스트 인식 결과 + confidence 추출 확인
상품명 / 수량 / 가격 라인이 실사용 가능한지 체감 평가

목적:
→ 이 프로젝트를 끝까지 가져갈 수 있는지 기술적 타당성 검증

3️⃣ OCR 출력 구조 고정 (LLM 입력용)
OCR 결과를 “조리된 구조화 텍스트 데이터” 형태로 통일
후보 값 + confidence를 포함한 구조 확정
이 구조를 LLM 입력 계약(contract) 으로 고정

목적:
→ OCR ↔ LLM 사이 중간 스키마 고정

4️⃣ LLM 프롬프트 설계 및 JSON Draft 확정
OCR 구조 데이터를 그대로 LLM 입력으로 사용
LLM이 최종 Receipt Draft JSON 생성하도록 프롬프트 설계
storeName / transactionDate / totalAmount / currency / paymentMethod / items / ocrConfidence
필드 확정

목적:
→ 프론트 · DTO · 엔티티 설계의 기준 스키마 확정

5️⃣ 프론트 Page 1 (영수증 등록 화면) 완성
파일 업로드
OCR → LLM 파이프라인 결과 표시
사용자 수정 UI 구성
“등록” 버튼까지 전체 흐름 완성
DB 저장은 아직 구현하지 않음

목적:
→ 입력 및 검증 UX 완성

6️⃣ OCR + LLM 결과 품질 최종 판단
“사람이 한 번 고치면 사용할 수 있는 수준인지” 기준으로 평가
통과 시 다음 단계 진행
미달 시 OCR 구조 또는 프롬프트 재조정

목적:
→ 품질 게이트 (이 단계 통과 못 하면 이후 단계 무의미)

7️⃣ CI/CD 파이프라인 구축
소스 push 시 build 자동 실행
(선택) dev 환경까지 자동 배포
운영 배포까지는 필수 아님

목적:
→ 실무 개발 흐름 및 자동화 경험 증명

8️⃣ 엔티티 / DTO 설계
확정된 Receipt Draft JSON 기준으로 설계
Receipt / ReceiptItem 최소 필드만 정의
DTO ↔ Entity 책임 분리 원칙 적용

목적:
→ 저장 구조 안정화 및 확장 대비

9️⃣ DB 저장 API 구현
Draft JSON → DB 저장
트랜잭션 처리
저장 성공/실패 응답 처리

목적:
→ 영구 데이터 관리 시작

🔟 히스토리 / 통계 페이지 구현
일자별 영수증 조회
날짜 필터 및 합계 표시
엑셀 추출 기능
프린트 기능

목적:
→ 관리 및 활용 화면 완성

✅ 최종 요약 문장 (설계 선언)
“폴더 구조와 책임을 먼저 고정한 뒤,
OCR 정확도를 검증하고,
OCR → LLM 파이프라인을 안정화한 후
CI/CD, DB 저장, 히스토리 화면을 순차적으로 완성한다.”
