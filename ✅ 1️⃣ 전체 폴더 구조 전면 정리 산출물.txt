✅ 1️⃣ 전체 폴더 구조 전면 정리
산출물 체크
 receipt-frontend / receipt-backend / receipt-ocr 분리
 프론트 pages / components / api / types 폴더 생성
 백엔드 controller / service / client / dto / domain / repository / config 폴더 생성
 OCR app / ocr / extract / schema / storage(temp, result) 폴더 생성
 각 폴더에 README 또는 주석으로 역할 설명 존재
 “누가 OCR을 하고, 누가 JSON을 만드는지” 구조상 명확
✅ 이 단계 통과 기준
→ 코드 없어도 구조만 보고 전체 흐름 설명 가능

✅ 2️⃣ OCR 교체 → PaddleOCR 적용
산출물 체크
 PaddleOCR 정상 실행
 실제 영수증 이미지 여러 장 테스트
 문자열 인식 결과 출력 가능
 숫자(금액, 수량) 인식 확인
 텍스트 단위 confidence 값 출력 확인
 OCR 실패/깨지는 케이스 확인
✅ 이 단계 통과 기준
→ “LLM으로 커버하면 쓸 수 있다”는 판단 가능

✅ 3️⃣ OCR 출력 구조 고정 (LLM 입력용)
산출물 체크
 OCR 결과를 구조화된 텍스트 데이터 형태로 통일
 상점/날짜/금액/상품이 “후보 구조”로 표현됨
 confidence가 포함됨
 OCR 결과가 storage/result/*.json 파일로 남음
 OCR 출력 구조 문서화(README 또는 주석)
✅ 이 단계 통과 기준
→ OCR 출력이 LLM에 바로 넣을 수 있는 입력 계약이 됨

✅ 4️⃣ LLM 프롬프트 설계 및 JSON Draft 확정
산출물 체크
 OCR 구조 데이터를 그대로 LLM 입력으로 사용
 LLM이 최종 Receipt Draft JSON 생성

 JSON 필드 확정
storeName
transactionDate
totalAmount
currency
paymentMethod
items(name, quantity, price)
ocrConfidence
 JSON 키 이름 고정
 잘못된 OCR에서도 “그럴듯한 JSON” 생성 확인
✅ 이 단계 통과 기준
→ 프론트·DTO·엔티티 설계 기준이 흔들리지 않음

✅ 5️⃣ 프론트 Page 1 (영수증 등록 화면) 완성
산출물 체크
 파일 업로드 UI 구현
 업로드한 이미지 미리보기
 OCR → LLM 결과 화면 표시
 상점명/날짜/금액/통화/결제수단 수정 가능
 상품 리스트(name/quantity/price) 수정 가능
 “등록” 버튼 존재
 DB 저장은 아직 안 함
✅ 이 단계 통과 기준
→ 사용자 입장에서 “이거 서비스 같네” 느낌 듦

✅ 6️⃣ OCR + LLM 결과 품질 최종 판단 (게이트)
산출물 체크
 상점명/금액/날짜가 대체로 맞음
 상품 라인이 대부분 의미 있음
 confidence 낮은 경우 구분 가능
 사람이 한 번 고치면 실사용 가능
 무조건 다시 OCR 해야 하는 수준 아님
✅ 이 단계 통과 기준
→ DB 단계로 넘어가도 괜찮다고 판단
❌ 통과 못 하면
→ 2~4단계로 돌아감

✅ 7️⃣ CI/CD 파이프라인 구축
산출물 체크
 Git push 시 build 자동 실행
 프론트 / 백엔드 최소 1개라도 자동 빌드 확인
 OCR 서버 Dockerfile 정상 빌드 가능
 실패 시 로그 확인 가능
✅ 이 단계 통과 기준
→ “실무 개발 흐름을 이해하고 있다” 증명 가능
